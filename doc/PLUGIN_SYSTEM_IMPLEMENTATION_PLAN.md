# YiCAD 新插件系统分步实施计划

本文是 [`PLUGIN_SYSTEM_DESIGN.md`](PLUGIN_SYSTEM_DESIGN.md) 的可执行实施计划。任务被设计为可分别交给不同 AI 执行，默认按编号顺序合并。

## 方案变更：旧插件机制先行废弃

- 新插件系统是对旧插件系统的直接替换，不是并行实现。现有基于 `QPluginLoader`、`PluginInterface`、`Document_Interface` 和 `Database_Interface` 的 Qt/C++ 插件 ABI 必须先移除，再开始新系统实现。
- 不提供旧插件二进制、源码接口、目录扫描方式或运行时行为的兼容层；不增加旧机制回退、双加载、功能开关或适配器。
- 已有旧插件 DLL 不能继续加载，插件作者必须基于新 C ABI/SDK 重新构建并使用新 XML 清单部署。这是明确接受的破坏性变更。
- 任务 00 是所有新插件系统任务的前置条件。后续任务不得重新引入已删除的旧接口或旧加载路径。
- 如果 [`PLUGIN_SYSTEM_DESIGN.md`](PLUGIN_SYSTEM_DESIGN.md) 中仍有“新旧机制并存”或“保留旧 ABI”的描述，以本实施计划的不兼容替换原则为准；执行任务时不得按冲突描述恢复旧机制。

以后用户只需输入：

```text
完成任务 XX
```

例如 `完成任务 03`。AI 必须自行读取 `AGENTS.md`、`PLUGIN_SYSTEM_DESIGN.md` 和本文，定位对应任务并完成实现、验证与汇报，不得要求用户重复任务内容或补充常规提示词。

## AI 执行协议

收到“完成任务 XX”后，AI 必须遵守以下流程：

1. 完整读取仓库根目录的 `AGENTS.md`、`PLUGIN_SYSTEM_DESIGN.md` 和本文；两份插件文档冲突时，以本文“方案变更：旧插件机制先行废弃”为准。
2. 检查工作区状态，保留用户已有改动；不得覆盖、回退或删除与当前任务无关的修改。
3. 检查当前任务的前置条件。若前置任务已经实现，直接继续；若当前任务已经部分实现，应审查、补全并验证，不得重复创建平行实现。
4. 只实现当前任务的范围，不提前实现后续任务，不顺手重构无关代码。
5. 遇到普通实现细节时，按本文的“冻结技术契约”自行决策，不向用户提问。
6. 本计划不要求新增或运行自动化测试；不得为任务创建测试目录、测试可执行程序或 CTest 条目。
7. 完成代码修改后不得执行 CMake 配置、编译、构建或安装；只进行源码级静态检查和差异检查。
8. 不修改真实的 `C:\ProgramData\YiCAD\plugins`。开发调试如需插件发现目录，必须使用临时目录和可注入的发现路径；真实目录只留给“用户验收”。
9. 不提交 `build/`、生成 DLL、本机路径、密钥、证书或真实插件 XML。
10. 未经用户明确要求不得创建 Git 提交、推送或修改远程仓库。
11. 最终回复必须包含：完成内容、修改文件、实际执行的命令及结果、未完成或残余风险、用户验收步骤。
12. 如果确有外部阻塞，先完成所有不受阻部分，再准确说明阻塞条件；不得把一般技术困难当作阻塞。

## 冻结技术契约

除非用户以后明确要求修改设计，所有任务统一采用以下规则，AI 不得在执行过程中重新询问。

### ABI

- 动态库边界只使用 C ABI，不跨边界暴露 C++ 类、Qt、STL、异常或由另一侧释放的内存。
- ABI 结果类型使用固定宽度 `int32_t`，`0` 表示失败，`1` 表示成功；不跨 DLL 使用 C++ `bool`。
- ABI 版本、结构大小使用 `uint32_t`。
- `YiCadHostApi` 和 `YiCadPluginApi` 的前两个字段为 `structSize` 和 `abiVersion`，仅用于新 C ABI 自身的版本协商与演进兼容，不用于兼容旧 Qt/C++ 插件 ABI。
- Windows 调用约定显式定义为 `__cdecl`；导出名称保持：
  - `yicad_plugin_get_abi_version`
  - `yicad_plugin_init`
  - `yicad_plugin_shutdown`
- `YiCadPluginAbi.h` 必须可以被 C 和 C++ 翻译单元包含。
- `YiCadPluginSdk.h` 是仅依赖 ABI 头的 C++23 内联便利封装，不是动态库边界。
- ABI v1 从第一次发布起就声明命令、Ribbon、当前文档、添加直线、添加圆、重生成、自动缩放、导入注册和导出注册函数，防止任务 10 再破坏 v1 布局。
- 宿主传入的函数表在插件 shutdown 返回前保持有效。
- 插件元数据字符串至少在 shutdown 返回前有效；宿主在 init 成功返回后立即复制为宿主所有的 UTF-8 字符串。
- 注册函数的所有字符串参数由宿主在函数返回前复制；命令和文件回调及其 `userData` 在 shutdown 前由插件保证有效。
- 异常不得穿越入口函数或回调。官方 SDK 示例的入口和回调必须 `noexcept` 或内部捕获全部异常。

### 发现、路径和加载

- 生产环境只扫描字面路径 `C:\ProgramData\YiCAD\plugins` 第一层的 `*.xml`，不递归，不扫描其他目录。
- 开发调试和验收可以通过构造参数注入临时发现目录；生产调用点必须显式使用固定目录常量。
- 每个 XML 只声明一个 `<plugin dll="..."/>`。
- DLL 相对路径以 XML 所在目录为基准；图标相对路径以插件 DLL 所在目录为基准。
- 不展开环境变量。目标必须规范化为存在的绝对 `.dll` 文件。
- XML 按文件名确定性排序：先进行不区分大小写比较，相等时再进行区分大小写比较。
- Windows 使用绝对路径和安全依赖搜索标志加载 DLL，不允许当前工作目录参与插件依赖解析。
- 单个插件失败不得阻止后续插件加载。

### 注册、命令和 UI

- 插件注册在 init 期间进入临时事务；只有 ABI、init、插件元数据、插件 ID 和所有引用校验成功后才整体提交。
- 失败、异常、重复 ID 或部分注册错误必须整体回滚，不留下 DLL 回调指针。
- 插件唯一键为 `pluginId`；命令唯一键为 `(pluginId, commandId)`。
- 命令行中的插件命令规范名为 `pluginId/commandId`，Ribbon 内部仍通过二元键调用。
- Ribbon 注册必须引用同一插件已经注册的命令。
- 同名 Ribbon tab/group 复用已有容器，不重复创建。
- 导入扩展名移除前导点并转为小写；v1 中同一扩展名只允许一个导入器。
- 导出格式的外部规范名为 `pluginId/formatId`。
- 插件注册和回调默认只允许在 UI 主线程发生，v1 不承诺线程安全。

### 文档和生命周期

- `YiCadDocumentHandle` 是非拥有型 opaque handle。插件不得长期保存，每次命令执行时重新获取当前文档。
- 宿主每次使用 handle 前必须确认它仍对应一个已打开文档；不得直接信任插件传回的任意指针。
- 添加实体使用带历史记录的 `EntityTable::add()`，不得使用持久化专用的 `add_direct()`。
- 插件状态为 `Discovered -> Validated -> Loaded -> Initialized -> Active -> Shutdown`。
- shutdown 按成功初始化的逆序执行，且每个插件最多一次；shutdown 完成后才能释放 DLL。
- v1 不支持热加载、热卸载、插件管理 UI、禁用配置或插件加载日志。
- 应用中只能存在新 C ABI 插件运行时；不得保留或恢复 `QPluginLoader/PluginInterface` 旧机制、旧 ABI 或双加载路径。

## 任务 00：废弃并移除旧插件机制

### 前置条件

- 无代码前置任务，且必须在任务 01 之前完成。

### AI 必须完成

1. 删除 `ApplicationWindow::loadPlugins()`、两个 `execPlug()` 入口及其调用点，移除 `QPluginLoader`、`qobject_cast<PluginInterface*>` 和旧插件 Ribbon/Action 创建逻辑。
2. 删除旧插件 ABI 头和实现：`YiCAD/src/plugins/PluginInterface.h`、`DocumentInterface.h`、`DatabaseInterface.h`、`filter/PluginIOInterface.h/.cpp`。
3. 删除旧宿主桥接实现：`YiCAD/src/main/DocPluginInterface.h/.cpp` 和 `DbPluginInterface.h/.cpp`。
4. 从 `FileIO` 过滤器工厂中移除 `PluginIOInterface`，保留并验证内置 `FilterOcdIO` 路由；不得用空实现冒充旧插件过滤器。
5. 清理 CMake 中仅服务于旧插件机制的源文件收集、包含目录和依赖。若 `YiCAD/src/plugins/` 清空，则删除该目录及对应 CMake 分组。
6. 清理仅供旧插件目录发现使用的代码和注释；不能删除仍被应用其他功能使用的通用目录逻辑。
7. 更新仓库内说明旧 Qt 插件仍可用的文档或注释，明确旧插件必须迁移到后续任务提供的新 SDK；不得在本任务提前实现新运行时。

### 禁止事项

- 不保留旧头文件作为转发头、弃用声明或编译兼容壳。
- 不保留旧 DLL 扫描、加载、命令执行或 FileIO 回退路径。
- 不增加 `YICAD_ENABLE_LEGACY_PLUGINS` 一类兼容开关。
- 不提前实现 `plugin_runtime`、新 XML 发现、新 DLL 加载器或新 UI 适配层。

### AI 静态检查

- 全仓搜索确认生产代码中不再引用 `QPluginLoader`、`PluginInterface`、`Document_Interface`、`Database_Interface`、`Doc_plugin_interface`、`Database_plugin_interface` 和 `PluginIOInterface`。
- `ApplicationWindow` 不再扫描或加载旧插件 DLL，也不再保留旧插件对象回调入口。
- `FileIO` 仍保留内置 OCD 导入导出过滤器。
- CMake 不再收集已删除的旧插件目录和文件。

### 用户验收

- 启动应用时，旧 Qt 插件 DLL 不会被发现或加载。
- 不安装任何插件时，应用能正常启动，内置命令和 OCD 打开/保存入口保持可用。
- 仓库中不存在供旧插件继续编译的 Qt/C++ ABI 头或兼容适配器。

### 完成定义

- 旧插件机制的加载入口、公开 ABI、宿主桥接和 FileIO 接口均已移除；仓库只保留实现新插件系统所需的干净基线。

## 任务 03：原生 DLL 加载器

### 前置条件

- 任务 01 已完成。任务 02 不是本任务的代码前置条件。

### AI 必须完成

1. 实现 `NativePluginLoader.h/.cpp`。
2. 封装 DLL 加载、三个固定导出函数查找、函数指针类型转换和卸载。
3. 类型应为 RAII、不可复制、可移动；重复 unload 必须安全。
4. Windows 使用 `LoadLibraryExW` 等安全策略，绝对路径加载并包含 DLL 自身目录的依赖搜索，不使用现有 `QPluginLoader`。

### AI 验证

- 正确 DLL 能取得三个非空入口。
- 缺少任意入口时加载结果失败且句柄被释放。
- move 构造、move 赋值和重复清理语义正确。
- 失败 DLL 不影响同进程加载下一 DLL。

### 用户验收

- 确认代码没有使用 `QPluginLoader`。
- 确认加载 API 使用规范化绝对路径。
- 确认没有把当前工作目录加入 DLL 搜索路径。

### 完成定义

- Loader 可以安全管理一个原生插件 DLL 及其三个 ABI 入口。

### AI 必须完成

### 完成定义

- Registry 能以原子事务保存或丢弃一个插件的全部注册内容。

### 完成定义

- 一个 ABI 插件可通过 HostApi 安全操作当前文档的最小几何能力。

## 任务 06：PluginManager 完整加载与关闭流程

### 前置条件

- 任务 01—05 已完成。

### AI 必须完成

1. 实现 `PluginManager.h/.cpp`。
2. 串联 ManifestReader、NativePluginLoader、HostApi 和 PluginRegistry。
3. 实现完整状态机和每个插件的状态记录。
4. 检查 ABI 版本和结构大小，初始化输出结构后调用 init。
5. init 返回后立即复制并校验插件 ID、名称、版本；插件 ID 必须与注册事务中的 ID 一致。
6. 成功时提交事务并进入 Active；失败时回滚、按需要调用 shutdown、释放 DLL，并继续下一个清单。
7. 保存成功初始化顺序；`shutdownAll()` 逆序关闭且幂等。

### 用户验收

- 单插件失败不会阻止后续正常插件。
- 错误或重复插件不会进入 Active。
- init 失败后没有 Registry 残留。
- shutdown 每个插件最多一次，并发生在 DLL 释放之前。
- Manager 没有实现日志、管理 UI 或热卸载。

### 完成定义

- 在发现目录中放置多份清单后，Manager 能确定性加载所有合法插件并安全关闭。
